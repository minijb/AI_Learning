# 计划系统核心理念

> 分层披露文档 — 仅在需要深入理解计划原则时加载。

---

## 「计划优先」原则

来源：Boris Tane

> "在审查过的书面计划存在之前，绝不让智能体写代码。"

**为什么：**
- 需求理解偏差 → 返工成本远高于计划成本
- 缺少可验证的验收标准 → 智能体无法自我检查
- 进度不可追踪 → 人类无法快速判断状态
- 计划散落在聊天记录中 → 不可版本化、不可复用

**反模式：** 让智能体边想边写，逐段生成代码。这在简单任务中可能奏效，但在跨模块改动中必然导致架构不一致。

## 双模式计划体系

### Quick Plan（轻量计划）

**适用场景：**
- 单文件修改、小 bug 修复
- 步骤 ≤5 步，预计耗时 <30 分钟
- 不需要跨模块协调

**格式：** Markdown，约 30 行模板
**存储：** `docs/exec-plans/active/<计划名>.md`

### Exec Plan（完整执行计划）

**适用场景：**
- 跨模块改动、新功能开发
- 步骤 >5 步，预计耗时 >30 分钟
- 需要功能列表追踪

**格式：** Markdown 主计划 + JSON 功能列表
**存储：** `docs/exec-plans/active/<计划名>/`

## 渐进式披露三层索引

新计划系统采用三级加载：

| 层级 | 文件 | 内容 | 加载时机 |
|------|------|------|---------|
| 1. 一句话摘要 | `PLAN.md` / `PLAN_COMPLETED.md` | 计划名 + 类型 + 一句话目标 + 依赖关系 | 会话启动、创建新计划前 |
| 2. 计划文档 | `exec-plan.md` / `.md` | 完整技术方案、步骤、风险、验收标准 | 需要执行或评审具体计划时 |
| 3. 执行记忆 | `memory.md` / `progress.txt` | 文件产出、决策记录、当前进度 | 跨会话恢复或深入排查时 |

**原则：** 先看摘要（10 秒了解全貌）→ 需要时深入计划文档 → 排查时查阅记忆。

## JSON 功能列表护栏

核心原则：**功能列表的 description/steps 由人类定义，智能体只能更新 passes 状态。**

```json
{
  "features": [
    {
      "id": "F1",
      "category": "核心功能",
      "description": "用户登录",
      "steps": ["验证输入", "查询数据库", "返回 token"],
      "passes": false
    }
  ]
}
```

| 字段 | 可修改者 | 说明 |
|------|---------|------|
| `id`, `category`, `description`, `steps` | **仅人类** | 定义"做什么" |
| `passes` | **智能体** | 反映"做完了吗" |

验证脚本 (`plan-validate`) 会拒绝任何对非 `passes` 字段的修改。

## 验证哲学：成功沉默，失败输出

来源：OpenAI 团队实践

- **通过的测试** → 只输出一行摘要计数
- **失败的测试** → 详尽输出，含：
  - 具体哪个检查失败
  - 期望值 vs 实际值
  - **修复指令**（告诉智能体怎么修）
- **输出格式** → grep-friendly，含显式标记 `[ERROR]` `[WARN]` `[PASS]`

> 4000 行通过结果涌入情境窗口 = 噪声淹没信号。只报告失败。

## 计划评审清单

人工评审计划时应检查：

1. 目标是否单一明确（一个计划只做一件事）
2. 验收标准是否可验证（每个标准能用脚本检查）
3. 步骤是否有明确的依赖关系（标注顺序和并行可能性）
4. 回滚方案是否存在（出错时如何回到干净状态）
5. 影响文件是否列举完整
6. 功能列表 JSON 的 passes 初始值是否全为 false

## 计划版本化

- 所有计划存储在 `docs/exec-plans/`（项目内）
- 建议随代码一起 git 版本控制（非强制）
- 不在外部系统（Google Docs、Slack、Notion）存储计划
- 引用外部文档时，将关键内容摘录到计划中

**原则：** 智能体运行时不在项目本地的内容，等于不存在。
