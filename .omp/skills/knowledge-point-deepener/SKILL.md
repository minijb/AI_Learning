# Knowledge Point Deepener — 知识点深度剖析

> 从"会用"到"理解每一行"——对单个知识点做显微镜级分析。

---

## 核心定位

学习过程中遇到一个概念，教程讲得不够深。本 Skill 把这个概念放在手术台上，
从 7 个层次逐层剖析。

---

## 触发条件

- 用户说"详细讲讲 X"
- 用户说"X 的原理是什么"
- 用户说"深入分析一下 X"
- 用户在学习教程中遇到不理解的概念要求展开

---

## 七层剖析框架

> **各层详细标准见 `docs/layered-analysis.md`**，包含每层的输出标准、示例、深度等级决策表。以下为速查框架。

对每个知识点，按以下层次展开（由浅入深）：

```text
第 1 层: 直觉理解 — 用类比解释"它是什么"
第 2 层: 使用场景 — 什么时候用，什么时候不用
第 3 层: API 层 — 所有接口和参数详解
第 4 层: 行为契约 — 输入输出约束、异常语义、线程安全保证
第 5 层: 实现原理 — 核心算法/数据结构，用伪代码解释
第 6 层: 源码分析 — 引用真实源码关键片段（标注行号和版本）
第 7 层: 对比与边界 — 同类技术对比、性能特征、设计取舍
```

**不是每个知识点都需要 7 层**。根据知识点的复杂度和用户需求决定深入到哪一层。
标注当前到达的层数: `[深度: 第N层]`

> 关联学习计划: [如果有]

## 工作流

1. **确定目标知识点** — 确认用户要深入的具体概念
2. **判断深度等级** — 根据知识点复杂度建议合适的层数
3. **逐层生成内容** — 从第 1 层开始，每层都是完整可读的
4. **插入源码引用** — 如涉及具体实现，标注项目和版本
5. **输出到 deep-dives/** — 调用 docs-manager 脚本更新索引：
   ```bash
   python .omp/skills/docs-manager/scripts/update-index.py deep-dives
   ```

---

## 护栏规则

1. **不跳过层次** — 即使第 1 层看起来"太简单"，也要写（建立直觉）
2. **源码必须标注版本** — 引用的代码标注项目和 commit/version
3. **对比必须对称** — 对比至少 3 个维度（性能/易用性/生态/学习曲线）
4. **输出独立可读** — 不假设读者已经看过教程，深度探索自包含
5. **文档规范** — 输出的所有 Markdown 文件遵循 `.omp/skills/markdown/SKILL.md`。
