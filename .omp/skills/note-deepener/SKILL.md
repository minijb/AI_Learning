# 笔记深化技能

## 做什么

用户把零散笔记（链接、摘录、随手记的概念）放在 `drafts/` 目录。本技能将这些笔记转化为结构化的深度教程，输出到 `docs/deeper_userNote/<笔记名>/`。

这不是简单的"翻译成中文"或"重新排版"。核心价值是**补充笔记中缺失的深度层次**：

- 笔记中提到的概念只说"是什么"，补充"为什么这样设计"和"底层怎么实现"
- 笔记中引用了某篇文章的结论，补充文章中跳过的推导过程
- 笔记中提到了一个技术方案，补充替代方案及其取舍
- 笔记中的代码是伪代码，补充可运行的实现

## 什么时候触发

用户明确指向 `drafts/` 下的某份笔记并表达深化意图时。典型触发语：

- "深化这份笔记" / "deepen this note"
- "帮我深入分析 drafts/xxx.md"
- "把这个笔记展开成教程"
- "根据我的笔记写一份深度分析"

不建议在用户只是提到某个概念时自动触发——除非用户明确指向了 `drafts/` 下的文件。

## 工作流程

### 第一步：读取并分析笔记（5-10 分钟思考）

读取目标笔记文件。在读完全文后，**不要立即开始搜索**——先在脑中过一遍：

1. **识别主题** — 笔记的核心主题是什么？是单一技术（如"客户端预测"）还是复合领域（如"多人游戏网络架构"）？
2. **提取概念列表** — 笔记中提到了哪些关键概念、技术名词、引用的文章/工具？
3. **评估深度缺口** — 对每个概念，判断笔记覆盖到了哪个层次：
   - 第 1 层：只提了名字 —— 需要从零解释
   - 第 2 层：有简要说明但缺少原理 —— 需要补充推导
   - 第 3 层：有原理但缺少实现细节 —— 需要补充代码/算法
   - 第 4 层：有实现但缺少性能/边界分析 —— 需要补充数据和对比
4. **识别可扩展方向** — 有哪些技术上强相关但笔记未覆盖的主题？例如：
   - 笔记讲"客户端预测"，可扩展到"服务器协调"、"确定性回滚"、"GGPO"
   - 笔记讲"实体插值"，可扩展到"外推 vs 插值"、"快照压缩"
   - 扩展必须有**明确的技术关联**，不能是"它们都是网络技术"这种松散关联
5. **检查已有知识库** — 搜索 `docs/deep-dives/` 下是否有与笔记主题相关的已有教程。如果存在，在撰写时直接引用它们（如"关于 X 的底层原理，详见 [已有的 deep-dive]($PATH)"），避免重复劳动，同时为读者建立知识链接。

### 第二步：联网搜索

对每个概念和扩展方向，执行联网搜索。搜索策略按优先级：

1. **官方文档和 RFC** — 最权威的定义和规范
2. **知名技术博客/教程** — Valve Developer Wiki、Gaffer On Games、Red Blob Games 等
3. **开源实现** — GitHub 上的实际代码，理解真实世界的复杂度和取舍
4. **社区讨论** — Reddit、Hacker News 上的实践经验和陷阱
5. **论文** — 学术视角（仅当主题有对应研究时）

搜索时关注这些信息：
- 概念的精确定义和术语来源
- 算法/协议的逐步推导
- 不同引擎/框架的实际实现方式
- 性能数据和 benchmark
- 常见的实现陷阱和边界情况
- 与其他方案的对比如 tradeoff

**如果联网搜索不可用**：不要跳过这一步。用你对技术的已有理解完成内容，但必须诚实标注——在所有无法核实的断言后标注 `[推测]` 或 `[基于内部知识]`。不要编造 URL 或虚构引用。

### 第三步：规划章节结构

基于分析结果和搜索结果，规划输出章节。原则：

- **2-6 个章节**，太少说明深度不够，太多说明切分过碎
- 每个章节一个独立 .md 文件
- 章节顺序从基础到深入，从核心到扩展
- 最后一章必须是"总结与对比"或"进一步阅读"

典型章节结构参考（根据实际内容调整）：

```
<笔记名>/
├── README.md           ← 目录和快速导航
├── 01-核心概念.md       ← 笔记中最核心的技术，深入解释
├── 02-实现细节.md       ← 算法、数据结构、代码
├── 03-边界与陷阱.md     ← 什么情况下会失败？怎么处理？
├── 04-替代方案.md       ← 其他解决同一问题的方法及对比
└── 05-总结与延伸.md     ← 总结 + 进一步阅读资源
```

章节命名使用中文，文件名使用拼音或英文 slug（如 `01-core-concept.md`），`README.md` 中建立中英文对照。

如果笔记内容单一、深度缺口小，可以只用 2-3 个章节。如果笔记覆盖了多个强相关的子主题，可以增加到 6 章。不限制字数——每章写透为止。

逐个章节撰写。遵循以下写作标准：

**文件头部模板**（每个 .md 文件）：
### 第四步：撰写章节

逐个章节撰写。遵循以下写作标准：

**文件头部模板**（每个 .md 文件，放在 H1 标题之后、正文之前）：

```markdown
# <章节标题>

> 基于笔记: <原笔记文件名>
> 所属教程: <教程总标题>
> 章: <序号>/<总数>
```

**写作风格——必须遵循"直觉→推导→实践→关联"四层递进**：

每章的开头必须是**直觉理解**——用一个类比、一张文字描述的图、或一个场景来建立心智模型。不要一上来就列定义。好的直觉理解例子："ECS 是'横切'数据，OOP 是'竖切'对象"；差的例子："ECS 由 Entity、Component、System 三部分组成"。

层序：
1. **直觉理解**（必写）— 类比、场景、文字示意图。让读者在 30 秒内理解"这东西解决什么问题"。
2. **算法与实现**（核心）— 逐步推导，从朴素版本到优化版本。每一步解释 WHY 而不只是 WHAT。
3. **实践考量**（深化）— 边界情况、性能数据、不同引擎/语言的取舍。用表格呈现对比数据。
4. **练习与参考答案**（巩固）— 每个章节末尾设计 1-3 道练习帮助读者检验理解。练习紧跟参考答案，答案用 Obsidian 可折叠 callout (`> [!tip]- 练习 N 参考答案`) 包裹，让读者先独立完成再展开查看。代码题给出可运行参考实现，概念题给出要点总结。
5. **关联主题**（收尾）— 指向本教程的其他章节、已有的 deep-dives、或外部资源。

**代码示例要求**：
- 使用相关语言的语法高亮标注
- 代码必须是可运行的（至少逻辑完整，不依赖不存在的库）
- **每一行关键代码加行内注释解释 WHY**，不只是复述 WHAT：

  ```cpp
  // 好：解释意图
  tail_.store(t + 1, std::memory_order_release);  // 保证 buffer 写入对消费者可见

  // 差：复述代码
  tail_.store(t + 1, std::memory_order_release);  // store tail+1 with release
  ```

- 对比写法比单一写法更有价值——用"朴素实现 → 优化实现"的渐进式推导：

  ```cpp
  // V1: 全 relaxed —— x86 碰巧工作，ARM 上出错
  tail_.store(t + 1, std::memory_order_relaxed);

  // V2: acquire/release —— 所有架构正确
  tail_.store(t + 1, std::memory_order_release);
  ```

**表格使用场景**：
- 多方案对比（方案 A vs B vs C，各维度打分）
- 适用场景推荐（什么情况下用哪种技术）
- 性能数据汇总（带单位和测试条件）

**笔记问题的显式回答**：如果笔记中提出了具体问题（如"EnTT 和 Flecs 的区别在哪？"），在对应章节的开头显式标注"回答笔记问题 #N："，让读者一眼看到他们的疑问在哪里被解决。

**引用规范**：
- 所有联网搜索到的信息必须标注来源链接
- 在段落中用 `[来源: 描述](URL)` 标注，或在一节末尾集中列出
- 不要假装知道——如果某个细节无法从搜索中确认，标注为 `[推测]`
- 如果引用了 `docs/deep-dives/` 下的已有内容，使用相对路径链接：

  ```markdown
  关于内存序的底层原理，详见 [C++ 内存序深度分析](../../../docs/deep-dives/cpp-memory-order-game-engine.md)
  ```

## 常见模式

以下是笔记中常见的内容类型及对应的深化策略：

| 笔记内容类型 | 深化策略 |
|-------------|---------|
| 链接集合 + 简短摘录 | 阅读链接原文 → 提取核心论点 → 补充推导过程 → 对比不同链接的观点 |
| 概念列表（无解释） | 每个概念从零解释 → 找概念间的关联 → 构建知识图谱 |
| 代码片段（无上下文） | 还原完整可运行代码 → 解释每一行的目的 → 添加错误处理 → 展示输入输出 |
| 问题/困惑记录 | 将问题转化为"探索"章节 → 搜索答案 → 提供多种解释角度 |
| 对比表格（草稿） | 补充更多维度 → 添加实测数据 → 解释差异的根本原因 |
| 单一技术笔记 | 补充替代方案 → 添加适用场景决策树 → 找到生产环境的实践案例 |


## 质量检查

完成全部章节后，自检：

1. **笔记中的每个关键概念都有对应章节覆盖**，没有被遗漏的
2. **每个引用/数据都有来源**，没有凭空断言
3. **代码示例可运行**（至少逻辑完整，不依赖不存在的库）
4. **章节之间有逻辑递进**，不是零散文章的拼凑
5. **扩展内容与原始主题有明确的技术关联**，不是跑题
6. **README.md 提供了完整的导航**，读者可以按需跳读
7. **练习题有对应参考答案**，没有只出题不给答案的情况
8. **答案格式正确** — 使用 `> [!tip]- 练习 N 参考答案` 可折叠 callout 包裹，绝对禁止使用 HTML `<details>` 标签。Callout 内每行内容必须以 `> ` 开头，确保在 Obsidian 中正确渲染。
   - 多个参考答案之间必须留一个空行，否则 Obsidian 会将所有答案渲染为同一个折叠块
   - 表格前必须留一个空行，紧贴普通文本的表格在 Obsidian 中常常无法正确渲染
